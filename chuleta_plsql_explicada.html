⚡ Chuleta PL/SQL — Sintaxis

Bloc Anònim

[DECLARE
  nom_var [CONSTANT] tipus [NOT NULL]
          [:= expr | DEFAULT expr];]
BEGIN
  sentències;
  [EXCEPTION
    WHEN nom_excepcio THEN
      accions;]
END;
%TYPE: v_sal taula.columna%TYPE;
DECLARE i EXCEPTION són opcionals
Estructura base de PL/SQL. Té tres parts:
· DECLARE (opcional): on declares variables i cursors.
· BEGIN: el codi que s'executa.
· EXCEPTION (opcional): captura errors.
→ Java: equivalent al bloc try { } catch { }

Variables

-- Declaració
nom_var [CONSTANT] tipus [NOT NULL]
        [:= expr | DEFAULT expr];

-- Assignació (dins BEGIN)
nom_var := expressió;

-- Des de SELECT
SELECT col INTO nom_var
FROM taula WHERE ...;
Prefixes conv: v_ variables · c_ constants · p_ paràmetres
Tres maneres d'assignar valor:
· := assignació directa (com = a Java).
· DEFAULT valor per defecte en la declaració.
· SELECT INTO carrega el resultat d'una query. Si no retorna res → NO_DATA_FOUND. Si retorna >1 fila → TOO_MANY_ROWS.
· %TYPE hereta el tipus d'una columna de taula automàticament.

IF / ELSIF / ELSE

IF condició THEN
  sentències;
[ELSIF condició THEN
  sentències;]
[ELSE
  sentències;]
END IF;
NULL en condicions → avalua a NULL → tractat com FALSE
Condicional estàndard. Igual que Java però:
· S'escriu ELSIF (sense segona E).
· Es tanca amb END IF.
· Si la condició és NULL, es tracta com FALSE (compte amb variables no inicialitzades).
→ Java: if (...) { } else if (...) { } else { }

Bucle FOR (numèric)

FOR variable IN [REVERSE] inici..final LOOP
  sentències;
END LOOP;
· La variable es declara implícitament (INTEGER)
· Àmbit: només dins el FOR
· REVERSE: itera de final a inici
· inici i final no poden ser NULL
Bucle numèric amb rang fix.
· La variable del bucle no cal declarar-la, es crea automàticament com INTEGER.
· Amb REVERSE itera de major a menor.
· Els límits no poden ser NULL (llançaria error en temps d'execució).
→ Java: for (int i = inici; i <= final; i++) { }

Bucle FOR (sense rang fix)

FOR variable IN (SELECT ... FROM taula) LOOP
  sentències;
END LOOP;

-- O amb cursor declarat:
FOR v_reg IN cur_nom LOOP
  sentències;
END LOOP;
Alternativa al WHILE per recórrer cursors. OPEN, FETCH i CLOSE són automàtics.
· El registre es declara implícitament (no cal declarar-lo).
· Quan s'acaben les files, el bucle para sol.
· És la forma recomanada quan no necessites control manual del cursor.
→ Java: for (Objecte o : llista) { }

Cursor Implícit (SQL%)

AtributDescripció
SQL%FOUNDTRUE si l'última sentència ha afectat ≥1 fila
SQL%NOTFOUNDTRUE si no ha afectat cap fila
SQL%ROWCOUNTNº de files afectades
S'activa automàticament amb INSERT/UPDATE/DELETE/SELECT INTO
Oracle crea un cursor automàtic per cada DML executat.
· SQL%FOUND: TRUE si ≥1 fila afectada.
· SQL%NOTFOUND: TRUE si cap fila afectada (útil per detectar UPDATE que no canvia res).
· SQL%ROWCOUNT: quantes files han estat afectades.
Es consulta just després de la sentència DML, abans de qualsevol altra.

Cursor Explícit — OPEN / FETCH / CLOSE

DECLARE
  -- 1. Declarar el cursor
  CURSOR cur_nom IS
    sentència_SELECT;
  -- Variables per guardar les dades
  v_camp1 taula.columna%TYPE;

BEGIN
  -- 2. Obrir (executa el SELECT)
  OPEN cur_nom;
  -- 3. Llegir files amb FOR
  FOR v_reg IN cur_nom LOOP
    sentències;
  END LOOP;
  -- 4. Tancar el cursor
  CLOSE cur_nom;
END;
Atribut
%FOUNDTRUE si el darrer FETCH ha obtingut fila
%NOTFOUNDTRUE si el darrer FETCH NO ha obtingut fila
%ROWCOUNTNº de files llegides fins ara
%ISOPENTRUE si el cursor està obert
Quan una query retorna múltiples files, cal un cursor explícit. Cicle de vida obligatori:
1. DECLARE CURSOR IS SELECT: defineix la query (no s'executa encara).
2. OPEN: executa el SELECT i posiciona el punter abans de la 1a fila.
3. FETCH INTO: llegeix una fila i avança el punter.
4. FOR..IN cursor LOOP: itera automàticament fins que no hi hagi més files.
5. CLOSE: allibera recursos. Sempre s'ha de tancar.
→ Java: com iterar un ResultSet de JDBC

Records — %ROWTYPE

DECLARE
  CURSOR cur_nom IS SELECT ...;
  -- Registre associat al cursor
  v_registre cur_nom%ROWTYPE;

BEGIN
  OPEN cur_nom;
  FOR v_registre IN cur_nom LOOP
    -- Accés als camps
    v_registre.nom_camp;
  END LOOP;
  CLOSE cur_nom;
END;
%ROWTYPE crea un registre amb els mateixos camps que el SELECT del cursor
Accés: nom_registre.nom_camp
En lloc de declarar una variable per cada columna, %ROWTYPE crea un "objecte" amb tots els camps de cop.
· S'adapta automàticament a l'estructura del SELECT del cursor.
· Accés per nom de camp: v_registre.nom_camp.
→ Java: com un DTO/record generat automàticament a partir del ResultSet

FOR..IN Cursor (simplificat)

-- Amb cursor declarat:
DECLARE
  CURSOR cur_nom IS SELECT ...;
BEGIN
  FOR v_reg IN cur_nom LOOP
    v_reg.nom_camp;
  END LOOP;
END;

-- Sense declarar cursor (subconsulta):
BEGIN
  FOR v_reg IN (SELECT ... FROM taula WHERE ...) LOOP
    v_reg.nom_camp;
  END LOOP;
END;
OPEN, FETCH i CLOSE automàtics · el registre es declara implícitament · àmbit: dins el FOR
La forma més còmoda de recórrer un cursor.
· OPEN, FETCH i CLOSE automàtics: Oracle ho gestiona tot sol.
· El registre (v_reg) es declara implícitament amb %ROWTYPE.
· Pots passar la query directament sense declarar el cursor (subconsulta).
· Quan s'acaben les files, el bucle para sol.
→ Java: com un for-each sobre una llista de DTOs

Cursor amb Paràmetres

DECLARE
  CURSOR cur_nom(p_param tipus) IS
    SELECT ... FROM taula
    WHERE columna = p_param;
BEGIN
  -- Obrir donant valor al paràmetre:
  OPEN cur_nom(valor);
  -- O amb FOR:
  FOR v_reg IN cur_nom(valor) LOOP
    ...
  END LOOP;
END;
· Només paràmetres d'entrada (IN)
· Tipus sense mida: NUMBER, no NUMBER(6)
· Vàlids només dins el cursor
Cursor reutilitzable amb filtre dinàmic.
· Es declara amb paràmetres i s'obre passant-li valors concrets.
· Només admet paràmetres IN (entrada).
· Els tipus no porten mida: NUMBER sí, NUMBER(6) no.
· El paràmetre és visible només dins la query del cursor.
→ Java: com un PreparedStatement amb paràmetres

Procediment

CREATE [OR REPLACE] PROCEDURE nom_proc
  ([p_param1 [IN|OUT|IN OUT] tipus,
    p_param2 tipus := valor_defecte,
    ...])
IS|AS
  [declaració variables locals;]
BEGIN
  sentències;
  [EXCEPTION
    WHEN nom_excepcio THEN accions;]
END [nom_proc];
MODES DE PARÀMETRES
ModeDescripció
INEntrada (per defecte). Read-only
OUTSortida. Retorna valor
IN OUTEntrada i sortida
CRIDAR PROCEDIMENT
-- Des de bloc anònim:
BEGIN
  nom_proc([params]);
END;

-- Amb OUT: cal variable
DECLARE v_res NUMBER;
BEGIN
  nom_proc(p1, v_res);
END;
· No es pot cridar des d'una sentència SQL · OR REPLACE sobreescriu la versió antiga
Subprograma que executa accions però no retorna valor directament.
· Modes de paràmetres: IN (read-only, per defecte), OUT (retorna valor), IN OUT (entrada i sortida).
· Per rebre un OUT cal declarar una variable i passar-la.
· OR REPLACE: sobreescriu sense haver d'esborrar primer.
· No es pot cridar des d'una sentència SQL, només des de blocs PL/SQL.
→ Java: mètode void amb paràmetres de sortida simulats per referència

Funció

CREATE [OR REPLACE] FUNCTION nom_func
  ([p_param1 [IN] tipus,
    ...])
RETURN tipus_retorn
IS|AS
  [declaració variables locals;]
BEGIN
  sentències;
  RETURN expressió;
  [EXCEPTION
    WHEN nom_exc THEN
      RETURN valor;]
END [nom_func];
· Sempre ha de tenir RETURN
· Es pot cridar des de SQL
· Cridar: v_x := nom_func(params);
· Tractar sempre NO_DATA_FOUND
Com un procediment però retorna un valor amb RETURN.
· Ha de tenir sempre almenys un RETURN al cos.
· A diferència del procediment, es pot cridar des de SQL.
· Els paràmetres solen ser IN (OUT i IN OUT estan desaconsellats en funcions).
· Tractar sempre NO_DATA_FOUND si fa SELECT INTO internament.
→ Java: mètode amb tipus de retorn no void

Trigger de Sentència (Statement-level)

CREATE [OR REPLACE] TRIGGER nom_trigger
TIMING  EVENT1 [OR EVENT2 OR EVENT3]  ON nom_taula
[DECLARE declaracions;]
BEGIN
  sentències;
END [nom_trigger];
TIMING
BEFOREExecuta el trigger abans de la sentència DML
AFTERExecuta el trigger després de la sentència DML
INSTEAD OFExecuta el trigger en lloc de la sentència DML
EVENT
INSERTAl insertar una fila
UPDATE [OF col]Al actualitzar
DELETEAl eliminar una fila
· S'executa 1 cop per sentència (independentment de quantes files afecti)
· Predicats condicionals: INSERTING, UPDATING, DELETING → BOOLEAN TRUE/FALSE
· UPDATING('COLUMNA') comprova si s'actualitza una columna concreta
· No es permeten: COMMIT, SAVEPOINT, ROLLBACK dins el cos
Codi que s'executa automàticament en resposta a un INSERT/UPDATE/DELETE.
· S'executa 1 sola vegada per sentència, independentment de quantes files afecti.
· BEFORE: s'executa abans del DML (útil per validar). AFTER: després (útil per auditoria).
· Usa INSERTING/UPDATING/DELETING per saber quin event l'ha disparat.
· No es pot fer COMMIT, SAVEPOINT ni ROLLBACK dins el cos del trigger.
→ Java: com un @PrePersist / @PostUpdate de JPA

Trigger de Fila (ROW) + :OLD / :NEW

CREATE [OR REPLACE] TRIGGER nom_trigger
TIMING  EVENT  ON nom_taula
FOR EACH ROW
BEGIN
  -- Accés als valors antics i nous:
  :OLD.nom_columna  -- valor abans del canvi
  :NEW.nom_columna  -- valor després del canvi
END;
Sentència:OLD:NEW
INSERTNULLnou valor
UPDATEvalor anticvalor nou
DELETEvalor anticNULL
· FOR EACH ROW → s'executa 1 cop per cada fila afectada
· Sense FOR EACH ROW → trigger de sentència (per defecte)
Com el trigger de sentència però s'executa una vegada per cada fila afectada.
· :OLD conté els valors abans del canvi. :NEW els valors després.
· En un INSERT: :OLD és NULL (no hi havia res). En un DELETE: :NEW és NULL (ja no hi serà).
· Pots modificar :NEW en un BEFORE trigger per canviar el valor que es guardarà.
· Si un UPDATE afecta 1000 files, el trigger s'executa 1000 vegades.
→ Java: com un listener que rep l'entitat abans i després de cada canvi

Excepcions comuns

ExcepcióQuan
NO_DATA_FOUNDSELECT INTO no retorna cap fila
TOO_MANY_ROWSSELECT INTO retorna >1 fila
EXCEPTION
  WHEN NO_DATA_FOUND THEN accions;
  WHEN TOO_MANY_ROWS THEN accions;
  WHEN OTHERS THEN accions;
Captura errors al final del bloc BEGIN.
· NO_DATA_FOUND: SELECT INTO no ha trobat cap fila.
· TOO_MANY_ROWS: SELECT INTO ha retornat més d'una fila (usa cursor si en vols moltes).
· WHEN OTHERS: captura qualsevol excepció no controlada, sempre al final.
· Dins el handler pots usar SQLERRM per obtenir el missatge d'error.
→ Java: catch (NoResultException e) / catch (NonUniqueResultException e) / catch (Exception e)
